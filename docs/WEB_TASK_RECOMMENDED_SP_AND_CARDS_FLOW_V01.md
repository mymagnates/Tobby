# Web Flow: Task Recommended SP + SP Cards v0.1

## 1) Page Structure

### A. Records (Independent Page)
- Purpose: global historical search and cross-entity lookup.
- Includes entry point to Search SP.
- Not the primary place to execute assignment decisions.

### B. Task Detail (Primary SP Decision Surface)
- Main content: task info, comments, files, timeline.
- Right panel: `Recommended SP` list with quick actions.

### C. SP Cards (Independent Page)
- Purpose: manage saved SP business cards and owner notes.
- Owner-specific list; each PM/PO manages own saved cards.

## 2) Task Detail Right Panel Actions

Per SP recommendation:
1. `Contact`
2. `Request Quote`
3. `Assign`
4. `Save Card`

Post-assignment behavior:
- Once one SP is assigned, other candidates become closed/disabled for that task context.

## 3) SP Cards Behavior

- Save action creates owner-scoped snapshot record:
  - key: `(owner_id, sp_id)`
  - includes cached profile fields + owner note.
- PM/PO can add/edit private note per saved card.
- Card can be reused to:
  - open SP profile
  - assign to another task
  - initiate contact

## 4) Data Rules

1. Task recommendation data:
- generated by matching engine for each task (`task_sp_recommendations`).

2. Saved card data:
- stored in separate owner library (`sp_cards`) as snapshot copy.

3. Decoupling:
- updating SP profile does not overwrite owner note or tags in saved card.
- optional refresh action can update snapshot fields while preserving owner note.

## 5) Mobile Alignment
- iOS should follow same business logic and state rules.
- UI adaptation allowed (drawer/section layout instead of right panel).
